🔀 Git Workflow: Colaboração em Equipe

📖 20 min de leitura 🎯 Nível: Intermediário 💻 Git Flow • PRs • Code Review 🔧 Ferramentas essenciais

📌 Introdução

Trabalhar em equipe com Git pode ser desafiador. Conflitos de merge, branches desorganizadas e commits confusos são problemas comuns. Neste tutorial, vamos dominar os workflows mais eficientes para colaboração, desde o Git Flow tradicional até estratégias modernas como GitHub Flow e trunk-based development.

🎯 Objetivos

  • Entender diferentes estratégias de branching
  • Dominar o Git Flow e suas branches (develop, feature, release, hotfix)
  • Aprender a resolver conflitos de merge como um profissional
  • Praticar code review e pull requests
  • Escrever commits semânticos e mensagens significativas

⚠️ Erro Comum #1: Commits sem descrição ou descrições ruins

❌ Erro: Mensagens de commit inúteis 
git commit -m "fix"
git commit -m "atualizacao"
git commit -m "teste"
git commit -m "mudanças"
git commit -m "final"

Problema: Impossível saber o que foi alterado, dificulta revisão, debugging e geração de changelogs.

✅ Correção: Commits semânticos (Conventional Commits) 
# Estrutura: <tipo>(escopo): <descrição>

git commit -m "feat(auth): adiciona login com Google"
git commit -m "fix(cart): corrige cálculo de frete"
git commit -m "docs(readme): atualiza instruções de instalação"
git commit -m "style(header): ajusta espaçamento do menu"
git commit -m "refactor(api): extrai validação para serviço separado"
git commit -m "test(unit): adiciona testes para função de cálculo"
git commit -m "chore(deps): atualiza dependências de segurança"
git commit -m "perf(image): otimiza carregamento de imagens"

# Tipos comuns:
# feat: nova funcionalidade
# fix: correção de bug
# docs: documentação
# style: formatação, espaçamento, etc (sem mudança de código)
# refactor: refatoração que não altera funcionalidade
# test: adiciona ou corrige testes
# chore: tarefas de manutenção, builds, etc
# perf: melhoria de performance
💡 Dica: Use git commit -m "tipo: descrição" -m "Descrição detalhada" para mensagens mais completas.

⚠️ Erro Comum #2: Trabalhar diretamente na branch main/master

❌ Erro: Commits diretos na main 
git checkout main
git add .
git commit -m "feature nova"
git push origin main

Problema: Sem code review, risco de quebrar produção, histórico linear e confuso, difícil reverter.

✅ Correção: Usar branches de feature e Pull Requests 
# 1. Criar branch para a feature
git checkout -b feature/login-google

# 2. Trabalhar na branch
git add .
git commit -m "feat(auth): implementa login com Google"

# 3. Atualizar com a main (evita conflitos)
git fetch origin
git rebase origin/main

# 4. Enviar branch para o repositório remoto
git push origin feature/login-google

# 5. Abrir Pull Request no GitHub/GitLab
# 6. Após aprovação e merge, deletar a branch
main        ●─────────●─────────●─────────●
              \         \         \         \
feature       ●───●─────●         ●───●─────●
(hotfix)                        /
release       ●─────────────────●

⚠️ Erro Comum #3: Resolver conflitos de forma incorreta

❌ Erro: Ignorar ou resolver conflitos sem entender 
# Conflito no arquivo app.js
<<<<<<< HEAD
function saudacao() {
    return "Olá, usuário!";
}
=======
function saudacao() {
    return "Welcome, user!";
}
>>>>>>> feature/internacionalizacao

# Resposta errada: simplesmente escolher um lado sem analisar
git checkout --ours app.js  # Ignora as mudanças da outra branch

Problema: Pode perder funcionalidades importantes ou introduzir bugs silenciosos.

✅ Correção: Resolver conflitos com cuidado e testar 
# 1. Identificar o conflito
git status

# 2. Abrir arquivo com conflito e resolver manualmente
# Manter o melhor de cada versão ou combinar ambas
function saudacao(idioma = 'pt') {
    if (idioma === 'pt') {
        return "Olá, usuário!";
    }
    return "Welcome, user!";
}

# 3. Marcar como resolvido
git add app.js

# 4. Continuar o merge/rebase
git commit  # Para merge
# ou
git rebase --continue  # Para rebase

# 5. Testar a aplicação após resolver conflitos!
🛠️ Ferramentas úteis:

Use git mergetool para abrir uma interface visual de resolução de conflitos. Ferramentas como VS Code, GitKraken e Sourcetree têm ótimos resolvers.

⚠️ Erro Comum #4: Não usar .gitignore corretamente

❌ Erro: Commitar arquivos desnecessários 
# Commitar node_modules, .env, arquivos de build
git add .
git commit -m "commit gigante"
# Resultado: repositório enorme, arquivos sensíveis expostos

Problema: Repositório lento, arquivos de ambiente vazam, conflitos frequentes em arquivos gerados.

✅ Correção: Configurar .gitignore adequado 
# .gitignore - Exemplo para projeto Node.js/React
node_modules/
dist/
build/
.env
.env.local
*.log
.DS_Store
coverage/
.vscode/
.idea/

# Remover arquivos já commitados acidentalmente
git rm --cached node_modules/ -r
git rm --cached .env
git commit -m "chore: remove arquivos desnecessários do git"
💡 Dica: Use gitignore.io para gerar .gitignore específico para seu stack.

🚀 Evolução: Workflows de colaboração

1. Git Flow (Tradicional)

# Estrutura de branches
main           # produção
develop        # desenvolvimento
feature/*      # novas funcionalidades
release/*      # preparação para produção
hotfix/*       # correções urgentes

# Fluxo:
git checkout develop
git checkout -b feature/nova-funcionalidade
# trabalhar...
git checkout develop
git merge --no-ff feature/nova-funcionalidade
git branch -d feature/nova-funcionalidade

# Para release:
git checkout -b release/1.0.0 develop
# ajustes finais...
git checkout main
git merge --no-ff release/1.0.0
git tag -a 1.0.0 -m "Versão 1.0.0"
git checkout develop
git merge --no-ff release/1.0.0

2. GitHub Flow (Simplificado)

# Mais simples, ideal para deploy contínuo
main           # sempre pronta para deploy
feature/*      # branches curtas

# Fluxo:
git checkout -b feature/nova-feature
# trabalhar e commitar
git push origin feature/nova-feature
# Abrir Pull Request
# Após aprovação, merge para main
git checkout main
git pull origin main
git branch -d feature/nova-feature

# Deploy automático após merge na main

3. Trunk-Based Development (Avançado)

# Para times maduros e CI/CD forte
trunk (main)   # única branch ativa
# branches de curta duração (menos de 1 dia)

# Fluxo:
git checkout -b short-lived-branch
# trabalhar rapidamente
git commit -m "feat: pequena mudança"
git pull origin trunk --rebase
git push origin short-lived-branch
# PR rápido e merge
git checkout trunk
git pull origin trunk

# Características:
# - Feature flags para controlar lançamentos
# - Deploys frequentes (múltiplos por dia)
# - CI/CD robusto
🎯 Evolução:

Comece com GitHub Flow (mais simples), evolua para Git Flow quando precisar de releases programadas, e adote Trunk-Based Development quando tiver CI/CD maduro e times experientes.

💡 Dicas Avançadas

  • Rebase vs Merge: Use git rebase para manter histórico linear, git merge para preservar contexto de branches
  • Interactive Rebase: git rebase -i HEAD~3 para squash, reorder e editar commits antes do PR
  • Git Hooks: Use husky (Node.js) ou pre-commit hooks para validar código antes do commit
  • Signed Commits: Configure GPG para assinar commits e garantir autenticidade
  • Git Aliases: Crie aliases para comandos frequentes

Git Aliases úteis

# Configurar aliases
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.ci commit
git config --global alias.st status
git config --global alias.unstage 'reset HEAD --'
git config --global alias.last 'log -1 HEAD'
git config --global alias.tree 'log --graph --oneline --all'

# Uso:
git co feature/nova
git st
git ci -m "mensagem"
git tree  # visualização gráfica do histórico

📝 Exercício Prático

Simule um fluxo de trabalho em equipe:

  • Crie um repositório e convide um colega (ou simule com branches)
  • Crie uma branch feature/login e implemente um formulário de login
  • Outro dev cria feature/header e modifica o mesmo arquivo
  • Abra PRs, faça code review com comentários construtivos
  • Resolva conflitos de merge que surgirem
  • Faça squash dos commits antes do merge
  • Deploy da main e crie uma tag de versão
🎯 Desafio extra: Implemente um hook de pre-commit que verifica se as mensagens seguem o padrão Conventional Commits.
