⚡ Vite vs Webpack: Qual Build Tool Escolher?

📖 15 min de leitura 🎯 Nível: Intermediário 💻 Configuração • Performance 🚀 Dev Experience

📌 Introdução

Webpack dominou o ecossistema JavaScript por anos, mas Vite surgiu como uma alternativa mais rápida e moderna. Ambos são bundlers (empacotadores) que transformam seu código em arquivos otimizados para produção, mas têm abordagens fundamentalmente diferentes. Neste tutorial, vamos comparar ambos, entender suas forças e fraquezas, e aprender quando escolher cada um.

🎯 Objetivos

  • Entender como Vite e Webpack funcionam internamente
  • Comparar performance de desenvolvimento e produção
  • Aprender configurações essenciais para cada ferramenta
  • Identificar casos de uso ideais para cada um
  • Migrar projetos do Webpack para Vite quando necessário

⚠️ Erro Comum #1: Usar Webpack para projetos simples

❌ Erro: Configurar Webpack para um projeto pequeno 
// webpack.config.js - 50+ linhas para um projeto simples
const path = require('path');
const HtmlWebpackPlugin = require('html-webpack-plugin');

module.exports = {
    entry: './src/index.js',
    output: {
        path: path.resolve(__dirname, 'dist'),
        filename: 'bundle.js',
    },
    module: {
        rules: [
            { test: /\.css$/, use: ['style-loader', 'css-loader'] },
            { test: /\.(png|jpg)$/, type: 'asset/resource' },
            // ... mais regras
        ],
    },
    plugins: [new HtmlWebpackPlugin()],
    devServer: { port: 3000, hot: true },
    // ... muito mais configuração
};

Problema: Over-engineering, configuração complexa desnecessária, tempo perdido com setup.

✅ Correção: Usar Vite para projetos simples/médios 
# Criar projeto Vite em segundos
npm create vite@latest meu-projeto -- --template vanilla
cd meu-projeto
npm install
npm run dev

# Configuração zero! Funciona imediatamente.
# Para React:
npm create vite@latest meu-app -- --template react
# Para Vue:
npm create vite@latest meu-app -- --template vue

⚠️ Erro Comum #2: Configuração incorreta de alias nos imports

❌ Erro: Paths relativos confusos 
// Importações feias e frágeis
import Button from '../../../components/Button';
import utils from '../../../../utils/helpers';
import api from '../../../api/client';

Problema: Código difícil de manter, quebra facilmente ao mover arquivos.

✅ Correção: Configurar alias 
// vite.config.js
import { defineConfig } from 'vite';
import path from 'path';

export default defineConfig({
    resolve: {
        alias: {
            '@': path.resolve(__dirname, './src'),
            '@components': path.resolve(__dirname, './src/components'),
            '@utils': path.resolve(__dirname, './src/utils'),
            '@api': path.resolve(__dirname, './src/api'),
        }
    }
});

// webpack.config.js
module.exports = {
    resolve: {
        alias: {
            '@': path.resolve(__dirname, 'src/'),
            '@components': path.resolve(__dirname, 'src/components/'),
        }
    }
};

// Agora imports ficam limpos:
import Button from '@components/Button';
import utils from '@utils/helpers';
import api from '@api/client';

⚠️ Erro Comum #3: Ignorar a diferença entre dev e build

❌ Erro: Assumir que dev e build funcionam igual 
// Em desenvolvimento, Vite usa ES Modules nativos
// Em produção, Vite usa Rollup para bundling

// Problema: Código que funciona em dev quebra em produção
if (import.meta.env.DEV) {
    // Código que só funciona com ES Modules
    const data = await fetch('/api/data');  // Funciona em dev
    // Mas em produção, pode precisar de URL absoluta
}

Problema: Código que funciona localmente quebra após o build.

✅ Correção: Entender as diferenças e usar variáveis de ambiente 
// Vite - usa import.meta.env
const API_URL = import.meta.env.VITE_API_URL || '/api';

// Webpack - usa process.env
const API_URL = process.env.API_URL || '/api';

// Verificar ambiente corretamente
const isDev = import.meta.env.DEV;  // Vite
const isDev = process.env.NODE_ENV === 'development';  // Webpack

// Configuração de proxy para evitar CORS em desenvolvimento
// vite.config.js
export default defineConfig({
    server: {
        proxy: {
            '/api': {
                target: 'http://localhost:3000',
                changeOrigin: true,
            }
        }
    }
});

🚀 Evolução: Configurações essenciais

1. Configuração Básica Vite

// vite.config.js
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';

export default defineConfig({
    plugins: [react()],
    server: {
        port: 3000,
        open: true,
        hmr: {
            overlay: true,
        },
    },
    build: {
        outDir: 'dist',
        sourcemap: true,
        minify: 'terser',
        rollupOptions: {
            output: {
                manualChunks: {
                    vendor: ['react', 'react-dom'],
                },
            },
        },
    },
    optimizeDeps: {
        include: ['react', 'react-dom'],
    },
});

2. Configuração Webpack Otimizada

// webpack.config.js
const path = require('path');
const HtmlWebpackPlugin = require('html-webpack-plugin');
const MiniCssExtractPlugin = require('mini-css-extract-plugin');
const TerserPlugin = require('terser-webpack-plugin');

module.exports = (env, argv) => {
    const isProduction = argv.mode === 'production';
    
    return {
        entry: './src/index.js',
        output: {
            path: path.resolve(__dirname, 'dist'),
            filename: isProduction ? '[name].[contenthash].js' : '[name].js',
            clean: true,
        },
        module: {
            rules: [
                {
                    test: /\.(js|jsx)$/,
                    exclude: /node_modules/,
                    use: 'babel-loader',
                },
                {
                    test: /\.css$/,
                    use: [
                        isProduction ? MiniCssExtractPlugin.loader : 'style-loader',
                        'css-loader',
                        'postcss-loader',
                    ],
                },
                {
                    test: /\.(png|jpg|svg)$/,
                    type: 'asset/resource',
                },
            ],
        },
        resolve: {
            extensions: ['.js', '.jsx'],
            alias: {
                '@': path.resolve(__dirname, 'src'),
            },
        },
        plugins: [
            new HtmlWebpackPlugin({
                template: './public/index.html',
            }),
            ...(isProduction ? [new MiniCssExtractPlugin({
                filename: '[name].[contenthash].css',
            })] : []),
        ],
        optimization: {
            splitChunks: {
                chunks: 'all',
                cacheGroups: {
                    vendor: {
                        test: /[\\/]node_modules[\\/]/,
                        name: 'vendors',
                        chunks: 'all',
                    },
                },
            },
            minimize: isProduction,
            minimizer: [new TerserPlugin()],
        },
        devServer: {
            port: 3000,
            hot: true,
            open: true,
            historyApiFallback: true,
        },
        devtool: isProduction ? 'source-map' : 'eval-source-map',
    };
};
🎯 Evolução:

Comece com Vite para projetos novos e simples. Mantenha Webpack para projetos legados ou quando precisar de configurações muito específicas. Considere migrar gradualmente projetos Webpack para Vite para melhorar a DX (Developer Experience).

💡 Quando Escolher Cada Um

✅ Escolha Vite quando:

  • Projetos novos (startups, side projects)
  • Você quer desenvolvimento extremamente rápido
  • Usa frameworks modernos (React, Vue, Svelte)
  • Não precisa suportar navegadores antigos (IE11)
  • Valoriza simplicidade e zero configuração

✅ Escolha Webpack quando:

  • Projetos legados que já usam Webpack
  • Precisa de suporte a navegadores antigos (IE11)
  • Requer plugins específicos sem alternativa no Vite
  • Configurações de build extremamente customizadas
  • Times já têm expertise em Webpack

🚀 Migrando de Webpack para Vite

# 1. Instalar dependências do Vite
npm install -D vite @vitejs/plugin-react

# 2. Criar vite.config.js
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';

export default defineConfig({
    plugins: [react()],
    server: {
        port: 3000,
    },
});

# 3. Atualizar scripts no package.json
{
    "scripts": {
        "dev": "vite",
        "build": "vite build",
        "preview": "vite preview"
    }
}

# 4. Mover index.html para a raiz (Vite usa como entry point)
# 5. Atualizar imports (Vite suporta ES Modules nativos)
# 6. Testar e ajustar configurações

💡 Dicas Avançadas

  • Module Federation: Webpack tem suporte nativo, Vite precisa de plugins como @originjs/vite-plugin-federation
  • Build analysis: Use vite-bundle-visualizer ou webpack-bundle-analyzer
  • SSR: Ambos suportam, mas Vite tem experiência mais suave com frameworks como Nuxt ou SvelteKit
  • Environment variables: Vite usa VITE_ prefixo, Webpack usa process.env

Análise de Bundle com Vite

// vite.config.js
import { defineConfig } from 'vite';
import { visualizer } from 'rollup-plugin-visualizer';

export default defineConfig({
